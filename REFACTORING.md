# Рефакторинг ShapeEditor: вынос фигур в отдельную DLL

## В чём была задача

Изначально в проекте всё лежало в одной куче: и окно редактора, и сам класс фигуры `ShapeBase`, и все её наследники (`PolygonShape`, `EllipseShape`, `CompoundShape` и т.д.). Получалось так: чтобы изменить одну фигуру, надо пересобирать всё приложение. И нет никакой возможности добавить новую фигуру «снаружи» — только переписав исходники самого редактора.

Нужно было:

1. **Вытащить фигуры в отдельную библиотеку (DLL)** — чтобы они жили сами по себе.
2. **Сделать так, чтобы редактор подгружал эту DLL динамически** через `System.Reflection` — то есть программа сама находит файл `ShapesLibrary.dll`, открывает его и достаёт оттуда классы фигур.
3. **Главное приложение не должно напрямую ссылаться на код фигур** — оно не знает, какие именно классы лежат внутри плагина, и работает с ними через общие «контракты» (интерфейсы).
4. Сделать это под **Windows**, на **WPF**, чтобы продолжало рисовать как раньше.

Зачем это нужно по-простому: если завтра кто-то захочет добавить новую фигуру — например, звезду — ему достаточно собрать свою DLL по тому же шаблону и положить её в папку `plugins`. Перекомпилировать главное приложение не надо.

---

## Как было устроено всё разом

Раньше структура была примерно такая:

```
ShapeEditor/
  shapes/
    ShapeBase.cs        ← базовый класс
    PolygonShape.cs     ← многоугольники
    EllipseShape.cs     ← эллипсы и круги
    CompoundShape.cs    ← группы фигур
  MainWindow.xaml.cs    ← окно редактора, прямо создаёт new PolygonShape() и т.д.
```

`MainWindow` напрямую делал `new PolygonShape()`, `new EllipseShape()` — то есть знал про эти классы в лицо.

---

## Что я сделал: новая структура

Решение я разделил на **три проекта**:

```
ShapeEditor.Core/        ← общий «фундамент» (интерфейсы и ShapeBase)
ShapesLibrary/           ← собственно фигуры, превращается в ShapesLibrary.dll (плагин)
ShapeEditor/             ← главное приложение (WPF), грузит плагин через рефлексию
```

### 1. `ShapeEditor.Core` — общая основа

Это маленькая библиотека, которая нужна и приложению, и плагину. В ней лежит:

- **`ShapeBase`** — абстрактный базовый класс для всех фигур (со всеми общими свойствами: цвет, масштаб, угол поворота, точка привязки, вершины, рисование, сериализация в JSON и т.д.).
- **«Контракты» (интерфейсы)** — это договорённости о том, что должна уметь каждая фигура:
  - `IShapeView` — общие члены: масштаб, угол, координаты, рисование, сохранение и т.п. Через него приложение работает с любой фигурой, не зная её типа.
  - `IEllipseShape` — то, что специфично для эллипса: оси, фокусы.
  - `IPolygonShape` — для многоугольников: сегменты, углы, перестройка вершин.
  - `ICompoundShape` — для групп: дети, смещения.
- **`IShapeFactory`** — «фабрика»: интерфейс, который умеет создавать фигуру по строковому имени (например, `"PolygonShape"`, `"EllipseShape"`).
- **`ShapePluginContext`** — статическое хранилище, куда после загрузки DLL кладётся ссылка на фабрику. Всё приложение потом достаёт фигуры именно отсюда.
- **`CompoundShapeHost` / `ShapeCompositionHost`** — статические «крючки», через которые плагин может **спросить у приложения** про состояние UI (например, «эта фигура сейчас редактируется?»). Это нужно, чтобы плагин **не зависел напрямую от `MainWindow`** — он просто дёргает функцию, которую приложение ему поставило при старте.

Почему `ShapeBase` живёт здесь, а не в плагине: потому что приложение тоже должен иметь возможность писать `ShapeBase shape`. Если бы базовый класс был в плагине, то главное приложение зависело бы от плагина — а это запрещено по условиям задачи.

### 2. `ShapesLibrary` — плагин с фигурами

Это библиотека (DLL), в которой лежат конкретные реализации:

- `PolygonShape` — все многоугольники (прямоугольник, треугольник, трапеция, шестиугольник, произвольный многоугольник).
- `EllipseShape` — эллипсы и круги.
- `CompoundShape` — группы из других фигур.
- `ShapePluginEntry` — **точка входа** плагина. Это статический класс с методом `CreateFactory()`, который возвращает `IShapeFactory`. Именно его ищет приложение через рефлексию.

Каждый класс наследуется от `ShapeBase` (из `Core`) и реализует свой интерфейс (`IPolygonShape`, `IEllipseShape`, `ICompoundShape`).

`ShapesLibrary` ссылается **только на `ShapeEditor.Core`**. Про `MainWindow` он ничего не знает.

### 3. `ShapeEditor` — главное приложение (WPF)

В нём остался `MainWindow`, всё рисование, вся панель параметров и т.д. Ключевые правки:

- **Убран код фигур** — папка `shapes/` удалена, файлы перенесены в `ShapesLibrary/`.
- **Главное приложение НЕ ссылается на `ShapesLibrary` напрямую как на код.** В csproj я добавил «технический» reference `ReferenceOutputAssembly=false` — это значит «соберите этот проект перед моим, но классов оттуда я при компиляции не вижу».
- **`ShapeLoader.cs`** — новый класс, который и делает всю магию динамической загрузки:
  - Ищет файл `ShapesLibrary.dll` сначала в папке `plugins/` рядом с приложением, потом просто рядом с exe.
  - Загружает сборку через `Assembly.LoadFrom`.
  - Через рефлексию находит тип `ShapeEditor.ShapePluginEntry` и его метод `CreateFactory()`.
  - Вызывает метод, получает `IShapeFactory`, кладёт его в `ShapePluginContext.Factory`.
  - Если что-то пошло не так (DLL не найдена, метод не на месте) — корректно возвращает `false`, без падения.
- **`App.xaml.cs`** — при старте приложения первым делом вызывает `ShapeLoader.TryLoadShapesPlugin()`. Если плагин не загрузился — показывает понятное окно с предупреждением, но не падает.
- **Создание фигур** в `MainWindow` теперь идёт через `ShapeLoader.CreateRectangle()`, `ShapeLoader.CreateEllipseToolbarDefault()` и т.п. — а не `new PolygonShape()`. Под капотом вызывается фабрика, которую дал плагин.
- **Работа с фигурами** — везде, где раньше было `PolygonShape ps`, теперь стоит `IPolygonShape ps`. То же для `IEllipseShape`, `ICompoundShape`. То есть приложение работает не с конкретными классами, а с их «обещаниями».

## Как этим пользоваться

### Сборка

В Visual Studio открыть `ShapeEditor.slnx`. Стартовым проектом сделать **`ShapeEditor`** (не Core и не ShapesLibrary — это библиотеки). Нажать **Build → Rebuild Solution**.

При сборке `ShapeEditor` автоматически срабатывает специальная цель `CopyShapesPlugin` (написана в его csproj), которая копирует только что собранный `ShapesLibrary.dll` в две папки рядом с exe:

- `ShapeEditor/bin/Debug/net9.0-windows7.0/plugins/ShapesLibrary.dll` — основной путь поиска;
- `ShapeEditor/bin/Debug/net9.0-windows7.0/ShapesLibrary.dll` — запасной путь.

### Запуск

Запустить `ShapeEditor.exe` (либо F5 в студии). При старте оно ищет плагин, загружает его и работает как раньше.

Если файла `ShapesLibrary.dll` нет ни в `plugins/`, ни рядом с exe — приложение **не падает**, а показывает окошко с предупреждением и подсказкой, куда положить файл.

### Добавление новой фигуры извне

Чтобы добавить, скажем, звезду:

1. Создать новый Class Library проект (.NET 9, Windows, WPF).
2. Сослаться на `ShapeEditor.Core.dll`.
3. Создать класс `StarShape : ShapeBase, IPolygonShape` (или какой подходит интерфейс).
4. Реализовать абстрактные методы.
5. Свою точку входа `ShapePluginEntry.CreateFactory()` сделать так, чтобы она возвращала фабрику, которая по строке `"StarShape"` создаёт `new StarShape()`.
6. Собрать DLL и положить рядом с приложением (или в `plugins/`).

В текущей реализации `ShapeLoader` ищет файл с конкретным именем `ShapesLibrary.dll` — если планируется грузить несколько плагинов, его легко расширить, чтобы он перебирал все `.dll` в папке `plugins/`.

## Что было сложного

1. **`MainWindow` нельзя было трогать сильно** — там 4500+ строк кода, который активно дёргает свойства фигур (`.Scale`, `.AnchorPoint`, `.Vertices` и т.д.). Если бы интерфейсы `IEllipseShape`/`IPolygonShape`/`ICompoundShape` были «голыми» (только специфичные методы), пришлось бы переписать сотни мест. Поэтому я ввёл общий интерфейс **`IShapeView`** с членами, общими для всех фигур, и сделал все три «родовых» интерфейса наследниками от него. Концретные классы (`PolygonShape`, `EllipseShape`, `CompoundShape`) автоматически удовлетворяют контракту, потому что унаследованы от `ShapeBase` (где все эти члены публичные).

2. **Плагин не должен знать про `MainWindow`** — но раньше код плагина обращался к `MainWindow.IsEditingThisChild(...)`, `MainWindow.IsHighlightedChild(...)`, `Application.Current.MainWindow as MainWindow` и лез через рефлексию в поля `_allShapes`. Это сильно связывало плагин с приложением. Я ввёл два «крючка» (`CompoundShapeHost` и `ShapeCompositionHost`) в `Core`: плагин дёргает делегаты из них, а `MainWindow` в своём конструкторе устанавливает эти делегаты. Связь развязана.

3. **Загрузка ресурсов JSON** — раньше в `ShapeBase.LoadFromFile` был жёсткий `switch` по строкам типа `"PolygonShape" => new PolygonShape()`. Теперь это делает фабрика из плагина, и `ShapeBase` сам её дёргает через `ShapePluginContext`.

4. **CSProj и автокопирование DLL** — иначе после сборки `ShapeEditor.exe` не нашёл бы плагин. Поэтому в csproj приложения добавлен MSBuild-target, копирующий `ShapesLibrary.dll` в нужные места после каждой сборки.

---

## Итог одной фразой

Главное приложение теперь не зависит от кода фигур — оно работает только с интерфейсами из `ShapeEditor.Core`, а конкретные фигуры подтягивает из `ShapesLibrary.dll` во время запуска через `Assembly.LoadFrom`. Можно подменить DLL — и набор фигур поменяется, без пересборки самого редактора.
